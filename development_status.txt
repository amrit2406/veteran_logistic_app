### Phase 1.5: Remember Me
* **Goal**: Implement a secure Remember Me feature that remembers the users login preference and username without bypassing authentication.

  * `RememberMeSettings` model containing only `Username` and `RememberMe` flag

  * `AuthenticationRememberMeServiceCollectionExtensions` for DI registration of `IRememberMeService` as singleton
  * Updated `LoginViewModel`:
    - Added `RememberMe` property
    - Loads saved settings on startup and restores username and checkbox state when Remember Me was previously enabled
    - After successful authentication: saves username and flag when checked, clears saved settings when unchecked
    - Does not restore password, auto-authenticate, auto-navigate, or create a session on startup
  * Updated `LoginView.xaml` with Remember Me checkbox bound to `RememberMe`
  * Updated `ServiceCollectionExtensions` to call `AddAuthenticationRememberMe()`
  * Graceful handling of missing or corrupted settings files via logging and default values
* **Storage Rules**:
  * Persisted: Username, RememberMe flag only
  * Never stored: Password, PasswordHash, Salt, Token, Session, UserId, Role, AuthenticationState
* **Explicitly Excluded**:
  * Automatic login, session persistence, Financial Year Selection, authorization, permissions, logout, password reset, MFA, Windows Credential Manager, and cloud synchronization
* **Validation Status**: Build verified successfully with no errors. Application still requires password entry and manual login even when Remember Me is enabled.

### Phase 2.3 C: Threading Fix for Navigation DataContext Updates
* **Goal**: Fix InvalidOperationException when navigating from Add User back to Users screen due to cross-thread DataContext assignment.
* **Included**:
  * **Modified** `MainWindow.xaml.cs`: Wrapped DataContext assignment in `OnCurrentViewModelChanged` with `Dispatcher.Invoke()` to ensure UI thread access. The navigation event was being raised on a background thread (due to ConfigureAwait(false) in async operations), causing WPF to throw "The calling thread cannot access this object because a different thread owns it" when updating the DataContext.
* **Root Cause**: When AddUserViewModel.SaveAsync() completes successfully, it calls `_navigationService.GoBackAsync()`. The NavigationService raises the CurrentViewModelChanged event on the background thread where the async operation completed. MainWindow.OnCurrentViewModelChanged() attempts to set DataContext, which is a UI-owned property, causing a cross-thread exception and application crash.
* **Explicitly Excluded**:
  * Navigation architecture redesign, async/await pattern changes, ConfigureAwait usage changes, Shell redesign, Phase 2.4 functionality.
